02 · Label casing — arrival airport
CLI — guessed capitalisation
var arrivalInput = page.GetByLabel("Arrival Airport");
MCP — exact DOM text
var arrivalInput = page.GetByLabel("Arrival airport");

Both pass because GetByLabel is case-insensitive in Playwright .NET — but the MCP selector reflects reality. If the label ever becomes case-sensitive (e.g. after a aria-label exact-match change), the CLI test would silently diverge.

03 · Button name casing — trip type
CLI — Title Case assumed
var roundTripButton = page.GetByRole( AriaRole.Button, new() { Name = "Round Trip" }); await roundTripButton.ClickAsync(); var oneWayButton = page.GetByRole( AriaRole.Button, new() { Name = "One Way" }); await oneWayButton.ClickAsync();
MCP — Sentence case from snapshot
// Toggle: Round trip → back to One way await page.GetByRole(AriaRole.Button, new() { Name = "Round trip" }).ClickAsync(); await page.GetByRole(AriaRole.Button, new() { Name = "One way" }).ClickAsync();

GetByRole name matching IS case-sensitive by default in Playwright. Both pass here only because Exact is not set, triggering substring/normalised matching. The MCP selectors are correct; the CLI selectors are accidentally working.

04 · Intermediate assertions before submit
CLI — verifies field state mid-flow
Assert.Contains("CAI", await arrivalInput.InputValueAsync()); Assert.Contains("Steigenberger ALDAU Beach Hotel", await destinationInput.InputValueAsync()); Assert.Equal("3", await adultsSelect.InputValueAsync()); // then submit...
MCP — submits directly, asserts end state
// (no intermediate assertions) var requestTask = page.WaitForRequestAsync(...); await page.GetByRole(AriaRole.Button, new() { Name = "Search" }).ClickAsync();

CLI is more defensive — it catches autocomplete failures before the submit. MCP focuses on the integration contract (POST body + redirect). Neither is wrong; they test different failure modes.

When to use each
CLI approach
  • You already know the DOM structure, or you wrote the component yourself.
  • You need a quick screenshot or smoke test without loading MCP schemas.
  • Token budget is tight — saves ~3.5k per session.
  • CI environment where MCP server is not available.
MCP approach
  • Testing a UI you didn't build — exact labels and roles come from the live DOM.
  • The page has localised or dynamic content where guessing casing is risky.
  • You want an audit trail linking selectors to a timestamped snapshot.
  • Showcase or documentation where the inspection step is itself the story.
